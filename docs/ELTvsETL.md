# ELT vs ETL 技术对比与选型分析

## 1. 本质区别

| 维度 | ETL（Extract-Transform-Load） | ELT（Extract-Load-Transform） |
|------|-----------------------------|------------------------------|
| 转换位置 | 抽取层，独立计算引擎 | 目标库内部，利用目标库算力 |
| 数据入湖 | 仅清洗后的数据写入目标 | 全量原始数据先写入，后转换 |
| 典型工具 | Informatica、DataStage、Kettle | dbt、Spark SQL、Databricks |
| 数据溯源 | 中间层清洗后原始数据已丢失 | ODS 层保留全量原始数据，随时可回溯 |
| 开发方式 | Pipeline（Java/Scala/图形化） | SQL 模型（dbt YAML + SQL） |
| 血缘粒度 | 粗（作业级） | 细（模型级，dbt 自动追踪列级血缘） |

**一张图说明核心差异：**

```
ETL:
  源系统 ──[抽]──▶ ETL引擎（清洗/转换/聚合）──[写]──▶ 数据仓库
                                  ↑
                            在这里完成所有转换
                            原始数据不进入目标库

ELT:
  源系统 ──[抽+写]──▶ 数据湖（Delta Lake）
                                     │
                              ┌──────▼──────┐
                              │  ODS层（原始）│
                              └──────┬──────┘
                                     │ SQL转换
                              ┌──────▼──────┐
                              │ DWD主题层   │
                              └──────┬──────┘
                                     │ SQL聚合
                              ┌──────▼──────┐
                              │ DWM汇总层   │
                              └─────────────┘
                            转换在目标库内完成，原始数据完整保留
```

## 2. 适用场景对比

### ETL 适合的场景

| 场景 | 说明 |
|------|------|
| **源端性能受限** | 老旧 SAP ECC / Oracle 10g 无法承受复杂查询，只适合轻量抽取 |
| **需要前置脱敏** | 敏感字段（身份证、信用代码）在进入数据湖前就脱敏，防止原始数据泄露 |
| **传输带宽有限** | 源端到目标端网络带宽有限，在中间层过滤无用数据减少传输量 |
| **超大规模日志采集** | 日志数据量大但逻辑简单（JSON解析），独立采集集群更高效 |
| **强一致性事务** | 多表联合抽取需要保证原子性，中间引擎的事务控制更成熟 |

### ELT 适合的场景

| 场景 | 说明 |
|------|------|
| **复杂业务逻辑需要频繁迭代** | 矿井编码映射、煤质-订单关联规则经常调整，SQL 模型改起来比 pipeline 快 |
| **需要保留完整原始数据** | PI 时序数据量大，保留原始数据才能做多次不同规则的质量检测 |
| **团队以 SQL 为核心能力** | Python/SQL 工程师可以直接写 dbt 模型，不需要 Java/Scala 开发资源 |
| **需要细粒度血缘** | dbt 自动追踪列级血缘，每层转换透明可查 |
| **需要历史数据回溯** | 编码规则变更后，用 ELT 可以对全量历史数据重新运行一遍新规则 |
| **目标库算力充足** | Spark 集群 / ClickHouse / Doris 本身算力强，不额外消耗传输层资源 |

## 3. A 公司的场景判断

### 3.1 为什么 A 公司适合 ELT

**场景特征一：源端是结构化关系型数据库**

SAP-ERP、LIMS、OA 都是标准 SQL 数据库，源端数据结构清晰，不需要在抽取层做复杂的 JSON 解析或非结构化数据清洗。数据直接批量导出或 CDC 接入，不需要独立 ETL 引擎做预处理。

**场景特征二：核心工作在"关联"而非"抽取"**

A 公司数据治理最大的痛点不是"数据抽不出来"，而是：

```
矿井编码各系统不统一（M001 vs M01 vs 1001）
客户名称同一客户多个写法
煤质批次号（CHARG）和订单行项目之间的关联需要重新建立
```

这些全部是**业务逻辑层面的关联和映射**，不是数据抽取层面的问题。用 ETL 在中间层做这些逻辑会导致：一旦规则调整，需要重写整个 pipeline；而用 ELT，只需要改几行 SQL 或 dbt 模型。

**场景特征三：需要历史数据回溯**

A 公司的编码标准化不可能一夜之间完成，新标准上线后需要对 18 个月的历史数据重新应用新规则。ELT 模式下，原始数据全量保留在 ODS 层，回溯时直接对 ODS 层重跑 DWD 转换逻辑即可；ETL 模式下，历史数据在中间层已被丢弃，无法回溯。

**场景特征四：数据质量检测需要原始数据**

PI 时序数据的质量问题（异常突升、设备掉线）需要用原始波形数据来验证，而不是清洗后的聚合值。ELT 保留原始数据才能支持任意时刻的质量规则重跑。

### 3.2 哪些环节仍然需要 ETL 思维

ELT 不代表完全抛弃 ETL，在两个关键环节仍然需要 ETL 前置处理：

**1. CDC 写入前的 SMT 过滤（Source-Level）**

```
Debezium CDC 事件 ──▶ Kafka Connect SMT ──▶ Delta Lake ODS
                                      ↑
                               只保留 IS_DELETED='N' 的记录
                               对敏感字段做不可逆脱敏
                               这两步本质上是 ETL 的 Transform
```

这一步是必要的，因为 Delta Lake ODS 层不应该存储已被软删除的记录（浪费存储），也不应该存储明文的敏感字段。

**2. ODS 层数据脱敏（Security-Level）**

```
ODS 层 ──脱敏作业──▶ 脱敏后 ODS 副本 ──▶ DWD
                   ↑
            STCD1（信用代码）
            NAME1（客户名称明文）
            这两步在 ODS 写入后独立执行
```

这一步是出于安全合规考虑，防止 ODS 层被有权限的用户查询到敏感原始数据。

## 4. 架构中的实际分工

```
[数据源]  ──CDC──▶ [Kafka] ──SMT过滤/脱敏──▶ [Delta Lake ODS]
                                                  │
                                          （从这里开始 ELT）
                                                  │
                                     ┌────────────▼────────────┐
                                     │  ODS 层（原始数据）      │
                                     │  保留全量，追加写入      │
                                     └────────────┬────────────┘
                                                  │ dbt / Spark SQL
                                     ┌────────────▼────────────┐
                                     │  DWD 层（主题标准化）    │
                                     │  矿井编码统一            │
                                     │  客户主数据关联          │
                                     │  跨系统 JOIN             │
                                     └────────────┬────────────┘
                                                  │ dbt aggregate
                                     ┌────────────▼────────────┐
                                     │  DWM 层（汇总指标）      │
                                     │  产销日榜                │
                                     │  煤质月报                │
                                     └────────────┬────────────┘
                                                  │ Doris Load
                                     ┌────────────▼────────────┐
                                     │  OLAP 层（即席分析）      │
                                     └─────────────────────────┘
```

**ETL 段（数据进入湖之前）：**
- SMT 过滤（软删除记录不写入）
- 敏感字段脱敏（STCD1 等）

**ELT 段（数据进入湖之后）：**
- ODS → DWD：清洗、编码标准化、主数据关联
- DWD → DWM：聚合、预计算指标
- DWM → OLAP：物化视图刷新

## 5. dbt 在 ELT 中的角色

dbt（data build tool）是 ELT 范式最典型的工具，在 ELT 架构中承担转换编排层：

| dbt 能力 | 对 ELT 的价值 |
|---------|-------------|
| SQL 模型定义 | 业务逻辑用 SQL 表达，工程师无需学 pipeline 语言 |
| 增量模型（incremental） | 只需处理新增数据，无需全量重跑 |
| 快照（snapshots） | 支持 SCD Type 2 历史变化追踪 |
| 单元测试（schema tests） | 每列可定义非空/唯一/枚举值等质量断言 |
| 文档生成 | 自动从 SQL 注释生成数据字典 |
| 列级血缘 | dbt 解析 SQL 自动构建列级血缘图谱 |

**dbt 在 A 公司 ELT 流程中的位置：**

```
Delta Lake ODS
      │
      │ dbt run --select dwd_production_daily
      ▼
Delta Lake DWD  (矿井生产标准化表)
      │
      │ dbt run --select dwd_coal_quality
      ▼
Delta Lake DWD  (煤质检测标准化表)
      │
      │ dbt run --select dwd_sales_production_wide  ← 跨3个ODS源关联
      ▼
Delta Lake DWD  (产销一体化宽表)
```

## 6. 选型结论

| 判断维度 | 结论 |
|---------|------|
| A 公司核心痛点 | 数据孤岛、编码不一致、跨系统关联——这些全部是业务逻辑问题，不是数据采集问题 |
| 技术栈现状 | Python/SQL 为主，无 Java ETL 开发资源 |
| 数据特征 | 结构化关系型为主（SAP/OA/LIMS），时序数据（PI）需保留原始波形 |
| 演进需求 | 编码标准会变，历史数据需要回溯重算 |
| **最终选择** | **ELT 为主 + 极轻量 ETL 前置** |

```
主线：ELT（ODS → DWD → DWM → OLAP）
支线：ETL 仅用于 CDC 入湖前的 SMT 过滤和敏感字段脱敏
```

这不是"二选一"，而是根据数据处理环节的性质选择合适的工具——在数据进入湖之前用 ETL 控制质量和安全，进入湖之后用 ELT 释放 SQL 的灵活性来完成业务关联和转换。
